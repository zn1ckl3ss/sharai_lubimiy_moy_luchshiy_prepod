# Проект

Реализация комплексного DevSecOps пайплайна для развертывания отказоустойчивого веб-приложения с автоматизированными проверками безопасности, качества кода и системой мониторинга.

## Архитектура и стек технологий

* **Backend:** Python 3.11, FastAPI, Uvicorn
* **CI/CD & GitOps:** GitHub Actions
* **SAST & Линтеры:** Bandit (проверка безопасности), Ruff (быстрый линтер и форматир), Pylint
* **Тестирование:** Pytest, HTTPX
* **Контейнеризация:** Docker, Docker Compose, Docker Hub
* **Оркестрация:** Kubernetes (Манифесты: Namespace, Deployment, Service, Ingress)
* **Инфраструктура как код (IaC):** Terraform (провайдер `null_resource`, `remote-exec`)
* **Управление конфигурацией:** Ansible
* **Мониторинг:** Prometheus, Grafana
* **Целевая ОС:** Ubuntu 24.04 LTS (Хостинг 62yun)

---

## Описание пайплайна (CI/CD)

При каждом `git push` в ветку `main` в облаке GitHub Actions автоматически запускается цепочка шагов:

1.  **Checkout & Setup:** Склонировать код и развернуть окружение Python.
2.  **SAST Scan (Bandit):** Статический анализ кода на наличие уязвимостей и дыр в безопасности.
3.  **Linter & Formatter (Ruff / Pylint):** Проверка стиля кода, кавычек, отступов и PEP 8 стандартов.
4.  **Testing (Pytest):** Автоматический запуск юнит-тестов для проверки эндпоинтов.
5.  **IaC (Terraform):** Безопасное подключение к VPS по SSH с использованием GitHub Secrets, обновление пакетов сервера.
6.  **Deployment (Ansible):** Подключение к VPS, установка Docker/Docker Compose, создание конфигурационных файлов и запуск контейнеров в режиме `detached`.

## Подробно: Техническая реализация этапов

### 1. Этап GitOps (Проверки безопасности и качества)
В рамках построения DevSecOps-конвейера настроен статический анализ и контроль качества кода в изолированном контейнере GitHub Actions (`ubuntu-latest`):
* **SAST-сканирование (Bandit):** Выполняется рекурсивный анализ AST-дерева Python-кода (команда `bandit -r app/`). Утилита проверяет код на типичные уязвимости (например, использование небезопасных генераторов случайных чисел, жестко зашитые пароли, уязвимые SQL-запросы).
* **Линтинг и форматирование (Ruff & Pylint):** * `ruff check` выполняет мгновенный анализ кода на синтаксические ошибки и неиспользуемые импорты.
    * `ruff format --check` гарантирует, что весь код соответствует строгим стандартам форматирования PEP 8.
    * `pylint` производит глубокую проверку архитектурных метрик кода.
* **Модульное тестирование (Pytest):** Создан тестовый клиент `TestClient(app)` на базе `httpx`. Перед запуском тестов переменная окружения `PYTHONPATH` принудительно выставляется в значение `.`, чтобы изолированный запуск `pytest` корректно определял внутреннюю архитектуру пакетов.

### 2. Контейнеризация и Мониторинг (Docker & Prometheus/Grafana)
Для изоляции приложения и среды его выполнения разработан многослойный `Dockerfile`:
* В качестве базового слоя использован оптимизированный образ `python:3.11-slim`, что позволило минимизировать размер итогового контейнера и уменьшить поверхность потенциальной атаки.
* Сборка Docker-образа автоматизирована, готовый артефакт тегируется и отправляется в публичный реестр Docker Hub под именем `soowhyy/sharai-app:latest`.
* **Инструментация метрик:** В код FastAPI интегрирована библиотека `prometheus-fastapi-instrumentator`. При старте приложения (`@app.on_event("startup")`) запускается сборщик, который автоматически экспонирует системные и бизнес-метрики (время ответа, количество HTTP-запросов по кодам ответа, нагрузка на память) на защищенный эндпоинт `/metrics`.
* **Оркестрация локального окружения:** В файле `docker-compose.yml` описана локальная изолированная сеть (Bridge network), связывающая три контейнера. Контейнер `prometheus` раз в 5 секунд опрашивает контейнер `web-app:8000/metrics` и сохраняет временные ряды (Time Series) в свою базу данных. `grafana` подключается к Prometheus в качестве источника данных (Data Source) для вывода дашбордов.

### 3. Оркестрация для Kubernetes (k8s манифесты)
Подготовлен декларативный набор манифестов для развертывания в production-кластере:
* `namespace.yaml`: Создает изолированный контур `sharai-project` для разграничения ресурсов (RBAC и NetworkPolicies).
* `deployment.yaml`: Описывает состояние контроллера. Задано свойство `replicas: 2` (запуск двух идентичных подов приложения для обеспечения высокой доступности и балансировки). Поле `imagePullPolicy: Always` гарантирует, что K8s будет скачивать свежий образ из Docker Hub при каждом перезапуске. Применяется стратегия плавного обновления (RollingUpdate).
* `service.yaml`: Декларирует абстракцию `ClusterIP`, которая создает стабильный внутренний IP-адрес и распределяет входящий TCP-трафик с порта 80 на внутренние порты контейнеров (8000).
* `ingress.yaml`: Описывает правила маршрутизации внешнего трафика Layer 7 (HTTP) через Ingress-контроллер (например, Nginx Ingress) внутрь кластера к созданному сервису.

### 4. Инфраструктура как код (Terraform IaC)
Так как целевой VPS-сервер от хостинга `62yun` уже запущен на ОС Ubuntu 24.04 LTS, Terraform применен для декларативного управления конфигурацией существующей ноды:
* Использован провайдер `hashicorp/null` и ресурс `null_resource`, позволяющие абстрагироваться от конкретного облачного провайдера API.
* Авторизация на сервере происходит по протоколу SSH с аутентификацией по асимметричному приватному ключу, безопасно передаваемому из GitHub Secrets через зашифрованную переменную `var.ssh_private_key`.
* Блок `provisioner "remote-exec"` выполняет каскадное обновление индексов пакетов `apt-get update` и устанавливает системные зависимости (сертификаты, curl, gnupg), необходимые для стабильной интеграции Docker и Ansible.

### 5. Конфигурация и Деплой (Ansible)
Финальный этап доставки приложения (CD) реализован через Ansible Playbook:
* В файле `hosts.ini` динамически сопоставляется IP-адрес хоста и путь к сгенерированному SSH-ключу.
* Плейбук выполняет идемпотентные (повторяемые без изменения результата) задачи:
    1.  Проверяет доступность порта 22 (SSH).
    2.  Устанавливает пакеты `docker.io` и `docker-compose-v2` штатными средствами пакетного менеджера `apt`.
    3.  Декларативно создает структуру каталогов `/app/sharai/monitoring` на сервере с правами `0755`.
    4.  Используя встроенный модуль `copy` с параметром `content`, Ansible напрямую генерирует файлы конфигурации `docker-compose.yml` и `prometheus.yml` на удаленной машине, избегая нестабильного копирования файлов по протоколу SFTP/SCP через межсетевые экраны.
    5.  Через модуль `shell` выполняет оркестрацию: останавливает старые контейнеры, принудительно стягивает обновленный образ из Docker Hub (`docker compose pull`) и запускает стек мониторинга и веб-приложения в фоновом режиме (`docker compose up -d`).
